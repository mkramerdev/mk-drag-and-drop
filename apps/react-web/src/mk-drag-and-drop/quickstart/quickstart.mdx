# Quickstart

`mk-drag-and-drop` is headless. You provide the elements, styling, state, and final drop behavior. The package wires those elements into a drag runtime and reports what happened.

Each example has DOM and React tabs. The DOM tab uses `@mk-drag-and-drop/dom` directly. The React tab uses `DragProvider` and hooks from `@mk-drag-and-drop/react`, which adapt the same DOM runtime to React refs.

## Installation

Install the package for the surface you use.

```bash
pnpm add @mk-drag-and-drop/dom
```

```bash
pnpm add @mk-drag-and-drop/react
```

Most React app code imports from `@mk-drag-and-drop/react`. Non-React code imports from `@mk-drag-and-drop/dom`.

## 1. Make An Element Draggable

Start with one draggable element and no drop targets. This proves the drag lifecycle is wired up: the item can be picked up, the app-provided overlay follows the drag, and releasing it does not change app data.

Look for the stable `draggableId`. In DOM, `createDraggable` registers an existing element. In React, `useDraggable` returns props and a ref for the rendered element.

## 2. Add One Drop Target

Next, register the destination as a droppable target. A successful drop calls `onDrop` with `dropTargetId`, then your app decides what to do.

The DOM tab moves the DOM node in the callback. The React tab updates state and rerenders. The package does not move the item in your application model by itself.

Only the destination is droppable in this step, so the item can move there but cannot move back yet.

## 3. Make Both Containers Valid

Now register both containers as drop targets and interpret both ids in `onDrop`. This is the same ownership model as before: the package reports a valid target, and your app commits the result.

This example also adds two common refinements:

- `onDragStart` and `onDragEnd` apply a ghost style to the original item while the overlay is being dragged.
- `maxOverlayCenterDistanceToRect({ maxDistance: 0 })` accepts a target only when the overlay center is inside that target.

Those refinements are optional. The important step is registering every container that should be a valid drop target.

## What To Remember

- Draggables and drop targets are connected by stable string ids.
- The package owns drag behavior, targeting, overlays, and lifecycle events.
- Your app owns rendering, state, persistence, and the final data commit.
- Use `onDrop` for successful commits and `onDragEnd` for cleanup that must run after every drag.
