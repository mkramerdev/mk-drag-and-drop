# React Sortable

React sortable behavior is for reorderable lists, columns, boards, and other UI
where items need to move relative to each other. Use `useSortable` when a
rendered element should be both a drag source and a drop target.

Use `useDraggable` and `useDroppable` when you only need to drag one item onto a
target and do not need sortable preview movement, before/after placement, or
sortable placement data.

The React package wraps the DOM package behavior with `DragProvider` and hooks. A
sortable item participates in the same provider scope, groups, targeting,
lifecycle callbacks, overlays, modifiers, and remeasurement behavior as the rest
of `@mk-drag-and-drop/react`. Your React components own markup, styling, state,
and final data commits.

## Minimal Sortable List

```tsx
import { useState, type ReactElement } from "react";
import {
  DragProvider,
  useSortable,
  type SortableDropPlacement,
} from "@mk-drag-and-drop/react";

type Item = {
  id: string;
  label: string;
};

const initialItems: Item[] = [
  { id: "task-1", label: "Write outline" },
  { id: "task-2", label: "Review API" },
  { id: "task-3", label: "Publish notes" },
];

export function SortableList(): ReactElement {
  const [items, setItems] = useState(initialItems);

  return (
    <DragProvider
      onDrop={({ draggableId, sortablePlacement }) => {
        if (!sortablePlacement) {
          return;
        }

        setItems((currentItems) =>
          moveItemToSortablePlacement(
            currentItems,
            draggableId,
            sortablePlacement
          )
        );
      }}
    >
      <div>
        {items.map((item) => (
          <SortableItem key={item.id} item={item} />
        ))}
      </div>
    </DragProvider>
  );
}

function SortableItem({ item }: { item: Item }): ReactElement {
  const sortable = useSortable<HTMLDivElement>({
    draggableId: item.id,
    group: "tasks",
  });

  return <div {...sortable}>{item.label}</div>;
}

function moveItemToSortablePlacement(
  items: readonly Item[],
  draggableId: string,
  placement: SortableDropPlacement
): Item[] {
  const item = items.find((currentItem) => currentItem.id === draggableId);

  if (!item) {
    return [...items];
  }

  const withoutItem = items.filter(
    (currentItem) => currentItem.id !== draggableId
  );
  const insertIndex = getPlacementIndex(
    withoutItem.map((currentItem) => currentItem.id),
    placement
  );

  return [
    ...withoutItem.slice(0, insertIndex),
    item,
    ...withoutItem.slice(insertIndex),
  ];
}

function getPlacementIndex(
  ids: readonly string[],
  placement: SortableDropPlacement
): number {
  if (placement.targetDraggableId !== null && placement.side !== null) {
    const targetIndex = ids.indexOf(placement.targetDraggableId);

    if (targetIndex !== -1) {
      return placement.side === "after" ? targetIndex + 1 : targetIndex;
    }
  }

  if (placement.previousDraggableId !== null) {
    const previousIndex = ids.indexOf(placement.previousDraggableId);

    if (previousIndex !== -1) {
      return previousIndex + 1;
    }
  }

  if (placement.nextDraggableId !== null) {
    const nextIndex = ids.indexOf(placement.nextDraggableId);

    if (nextIndex !== -1) {
      return nextIndex;
    }
  }

  return ids.length;
}
```

## `useSortable` Options

`useSortable(options)` must be called inside `DragProvider`.

```ts
type UseSortableOptions = {
  draggableId: string;
  group?: string;
  containerId?: string | null;
  axis?: "vertical" | "horizontal";
  placementBoundary?: {
    start?: number;
    end?: number;
  };
};
```

Required options:

- `draggableId`: stable id for the sortable item. This also becomes the sortable item's drop target id.

Optional options:

- `group`: drag-and-drop group. Defaults to `"default"`.
- `containerId`: logical container containing the item. Defaults to `null`.
- `axis`: sortable axis used by placement behavior. Defaults to `"vertical"`.
- `placementBoundary`: reversal placement ratios. Defaults to `{ start: 0.25, end: 0.75 }`.

## `useSortable` Return Value

`useSortable<ElementType>()` is generic over the host element type and defaults to `HTMLElement`.

```ts
type UseSortableResult<
  ElementType extends HTMLElement = HTMLElement,
> = React.HTMLAttributes<ElementType> & {
  ref: React.RefCallback<ElementType>;
};
```

The returned props include `ref`. When pointer dragging is enabled, they include pointer activation handlers. When keyboard dragging is enabled, they include `tabIndex` and keyboard handlers.

Because the result is typed as `HTMLAttributes<ElementType> & { ref: RefCallback<ElementType> }`, spread it onto the rendered sortable element.

```tsx
const sortable = useSortable<HTMLArticleElement>({
  draggableId: card.id,
  group: "cards",
});

return <article {...sortable}>{card.title}</article>;
```

## Provider Setup

`DragProvider` creates one drag-and-drop scope for its React subtree. It coordinates registrations, targeting, lifecycle callbacks, overlays, modifiers, keyboard and pointer input, and remeasurement.

```tsx
<DragProvider
  onDragUpdate={({ activeDropTargetId }) => {
    setActiveTargetId(activeDropTargetId);
  }}
  onDrop={({ draggableId, sortablePlacement }) => {
    if (!sortablePlacement) {
      return;
    }

    setItems((items) =>
      moveItemToSortablePlacement(items, draggableId, sortablePlacement)
    );
  }}
>
  <SortableList />
</DragProvider>
```

Use `onDrop` for durable sortable order changes. `onDragEnd` runs before
`onDrop` for a successful drop and does not include `sortablePlacement`. Use
`onDragUpdate` for lightweight UI feedback such as active target styling.

## Targeting vs Placement

Targeting and sortable placement are separate.

### Targeting

Targeting chooses the active drop target from registered targets in the active group. The public signals are `DragUpdateEvent.activeDropTargetId` and `DropEvent.dropTargetId`.

### Placement

Placement decides before/after or container placement relative to the active
target. The public signal is `DropEvent.sortablePlacement`.

`dropTargetId` is the active target that won targeting. In a sortable board, that
target can be an item id or a container id. Use `sortablePlacement.containerId`
as the destination container when sortable placement is present.

### Lifecycle

Lifecycle callbacks let the app commit React state and clean up app UI through `onDragStart`, `onDragUpdate`, `onDragEnd`, and `onDrop`.

The provider's `targetingAlgorithm` chooses the active target. Sortable placement
does not choose the target, and `placementBoundary` is not a targeting threshold.

## Committing Order In React State

Sortable placement is exposed as:

```ts
type SortableDropPlacement = {
  sourceContainerId: string | null;
  containerId: string | null;
  previousDraggableId: string | null;
  nextDraggableId: string | null;
  targetDraggableId: string | null;
  side: "before" | "after" | null;
};
```

Field meanings:

- `sourceContainerId`: dragged item's container at drag start.
- `containerId`: destination container. This can be `null` for sortable sets that do not use containers.
- `previousDraggableId`: nearest sortable item before the final position, when available.
- `nextDraggableId`: nearest sortable item after the final position, when available.
- `targetDraggableId`: active sortable item anchor when the drop lands relative to an item.
- `side`: `"before"` or `"after"` relative to `targetDraggableId`, or `null` for container-only placement.

A practical single-list commit helper can prefer the exact active target anchor
first, then fall back to sibling anchors:

```tsx
<DragProvider
  onDrop={({ draggableId, sortablePlacement }) => {
    if (!sortablePlacement) {
      return;
    }

    setItemIds((ids) => reorderIds(ids, draggableId, sortablePlacement));
  }}
>
  {children}
</DragProvider>
```

```ts
function reorderIds(
  ids: readonly string[],
  draggableId: string,
  placement: SortableDropPlacement
): string[] {
  const withoutDragged = ids.filter((id) => id !== draggableId);
  let insertIndex = withoutDragged.length;
  let resolvedPlacement = false;

  if (placement.targetDraggableId !== null && placement.side !== null) {
    const targetIndex = withoutDragged.indexOf(placement.targetDraggableId);

    if (targetIndex !== -1) {
      insertIndex = placement.side === "after" ? targetIndex + 1 : targetIndex;
      resolvedPlacement = true;
    }
  }

  if (!resolvedPlacement && placement.previousDraggableId !== null) {
    const previousIndex = withoutDragged.indexOf(
      placement.previousDraggableId
    );

    if (previousIndex !== -1) {
      insertIndex = previousIndex + 1;
      resolvedPlacement = true;
    }
  }

  if (!resolvedPlacement && placement.nextDraggableId !== null) {
    const nextIndex = withoutDragged.indexOf(placement.nextDraggableId);

    if (nextIndex !== -1) {
      insertIndex = nextIndex;
    }
  }

  return [
    ...withoutDragged.slice(0, insertIndex),
    draggableId,
    ...withoutDragged.slice(insertIndex),
  ];
}
```

`sortablePlacement` can be omitted. Treat that as no sortable order change unless
your app has another reason to update state.

## Containers And Empty Lists

For one simple list, `containerId` can be omitted. For boards, lanes, columns, or
empty lists, register containers with `useDropContainer` and give sortable items
a matching `containerId`.

In multi-container sortables, treat `sortablePlacement.containerId` as the
destination and `sortablePlacement.sourceContainerId` as the drag-start source.
Remove the dragged id from every container first, then insert it into the
destination using `targetDraggableId`, `previousDraggableId`, and
`nextDraggableId`.

```tsx
import { useState } from "react";
import {
  DragProvider,
  useDropContainer,
  useSortable,
  type SortableDropPlacement,
} from "@mk-drag-and-drop/react";

type Lane = {
  id: string;
  cardIds: string[];
};

function Board({ initialLanes }: { initialLanes: Lane[] }) {
  const [lanes, setLanes] = useState(initialLanes);

  return (
    <DragProvider
      onDrop={({ draggableId, sortablePlacement }) => {
        if (!sortablePlacement) {
          return;
        }

        setLanes((currentLanes) =>
          moveCard(currentLanes, draggableId, sortablePlacement)
        );
      }}
    >
      {lanes.map((lane) => (
        <LaneView key={lane.id} lane={lane} />
      ))}
    </DragProvider>
  );
}

function LaneView({ lane }: { lane: Lane }) {
  const container = useDropContainer<HTMLDivElement>({
    containerId: lane.id,
    group: "cards",
  });

  return (
    <div {...container}>
      {lane.cardIds.map((cardId) => (
        <Card key={cardId} cardId={cardId} laneId={lane.id} />
      ))}
    </div>
  );
}

function Card({
  cardId,
  laneId,
}: {
  cardId: string;
  laneId: string;
}) {
  const sortable = useSortable<HTMLElement>({
    draggableId: cardId,
    group: "cards",
    containerId: laneId,
  });

  return <article {...sortable}>{cardId}</article>;
}

function moveCard(
  lanes: readonly Lane[],
  draggableId: string,
  placement: SortableDropPlacement
): Lane[] {
  if (placement.containerId === null) {
    return [...lanes];
  }

  return lanes.map((lane) => {
    const idsWithoutCard = lane.cardIds.filter(
      (cardId) => cardId !== draggableId
    );

    if (lane.id !== placement.containerId) {
      return { ...lane, cardIds: idsWithoutCard };
    }

    return {
      ...lane,
      cardIds: reorderIds(idsWithoutCard, draggableId, placement),
    };
  });
}
```

An empty container can receive a sortable item when its drop container is
registered. In that case placement may have `containerId` set while item anchors
such as `previousDraggableId`, `nextDraggableId`, `targetDraggableId`, and `side`
are `null`.

For non-empty destination containers, the active target may be a child sortable
item. For broad or empty destination regions, the active target may be the
container itself. In both cases, commit code should read the normalized
destination from `sortablePlacement.containerId`.

## Axis

Sortable supports two public axis values:

- `"vertical"`: lists, menus, and stacked rows.
- `"horizontal"`: columns, tabs, and horizontal boards.

The default axis is `"vertical"`.

```tsx
const row = useSortable({
  draggableId: rowId,
  group: "rows",
  axis: "vertical",
});
```

For horizontal sorting, set `axis: "horizontal"` on the sortable items in that
sortable group or container.

```tsx
const column = useSortable({
  draggableId: columnId,
  group: "columns",
  containerId: "board",
  axis: "horizontal",
});
```

Use one axis consistently for items that sort together.

## Placement Boundaries

`placementBoundary` tunes before/after switching when placement reverses over the same active target.

```tsx
const sortable = useSortable({
  draggableId: item.id,
  group: "items",
  placementBoundary: {
    start: 0,
    end: 1,
  },
});
```

The public type is:

```ts
type SortablePlacementBoundary = {
  start?: number;
  end?: number;
};
```

Defaults are:

```ts
{
  start: 0.25,
  end: 0.75,
}
```

The values are ratios of the active target on the sortable axis:

- `start`: used when movement is forward on the sortable axis. Placement can
  switch after the position crosses the start ratio.
- `end`: used when movement is backward on the sortable axis. Placement can
  switch before the position crosses the end ratio.

On first entry into a newly active target in the same container, sortable
placement remains movement-responsive: forward movement places after, backward
movement places before, and no movement falls back to the midpoint. When entering
a different DOM container, the initial side is midpoint-based using the current
sortable placement position. Pointer-based targeting uses the pointer as that
position; rect-based targeting such as `centerToCenter` uses the overlay center.
Boundaries are same-target reversal and hysteresis controls; they do not decide
the initial cross-container side or delay the first movement after midpoint-based
entry.

Compact rows often work well with defaults. Large targets may benefit from wider
hysteresis so placement does not flip too easily during small pointer reversals.

## Drag Handles With Sortable

Use `useDragHandle` to restrict dragging to a child handle.

```tsx
import {
  useDragHandle,
  useSortable,
} from "@mk-drag-and-drop/react";

function Row({ id, label }: { id: string; label: string }) {
  const sortable = useSortable<HTMLDivElement>({
    draggableId: id,
    group: "rows",
  });
  const handle = useDragHandle<HTMLButtonElement>();

  return (
    <div {...sortable}>
      <button {...handle} type="button" aria-label={`Drag ${label}`}>
        Drag
      </button>
      <span>{label}</span>
    </div>
  );
}
```

Use handles when the sortable item contains buttons, links, inputs, or selectable content.

## Drag Overlays With Sortable

The current React API does not export a public `DragOverlay` component. Use the `dragOverlay` prop on `DragProvider`.

```tsx
<DragProvider
  dragOverlay={({ dragState }) => (
    <div className="dragOverlay">{dragState.draggableId}</div>
  )}
>
  {children}
</DragProvider>
```

Overlay movement is handled by the package. App code should not update React state on every pointer move just to move the overlay.

For release animations, set `overlayRelease="manual"` and call `removeOverlay` when the released overlay should be removed.

```tsx
<DragProvider
  overlayRelease="manual"
  dragOverlay={(input) => (
    <div
      className={
        input.phase === "released"
          ? "overlay overlayReleased"
          : "overlay"
      }
      onAnimationEnd={
        input.phase === "released" ? input.removeOverlay : undefined
      }
    >
      {input.dragState.draggableId}
    </div>
  )}
>
  {children}
</DragProvider>
```

`removeOverlay` is only present for the `"released"` phase. If overlay-local UI
changes visual bounds during a sortable drag, call `input.remeasureOverlay()`
from the `dragOverlay` render input.

## Remeasurement

Drop targeting uses measured target rectangles. The package measures targets when
they are registered and at drag start. During normal sortable preview movement,
app code does not need to call remeasurement just to move the preview.

Call `useRemeasureDropTargets` after app-owned layout changes that should affect
targeting during a drag. Common examples are expanding rows, collapsing sections,
loading more items, changing item heights, or showing and hiding nested drop
targets.

```tsx
import { useLayoutEffect, useState } from "react";
import {
  useRecomputeActiveDrag,
  useRemeasureDropTargets,
} from "@mk-drag-and-drop/react";

function ExpandableSection({ group }: { group: string }) {
  const [expanded, setExpanded] = useState(false);
  const recomputeActiveDrag = useRecomputeActiveDrag();
  const remeasureDropTargets = useRemeasureDropTargets();

  useLayoutEffect(() => {
    requestAnimationFrame(() => {
      remeasureDropTargets({ group });
      recomputeActiveDrag();
    });
  }, [expanded, group, remeasureDropTargets, recomputeActiveDrag]);

  return (
    <button
      type="button"
      onClick={() => setExpanded((value) => !value)}
    >
      {expanded ? "Collapse" : "Expand"}
    </button>
  );
}
```

Supported inputs are:

```ts
remeasureDropTargets();
remeasureDropTargets("target-id");
remeasureDropTargets(["target-a", "target-b"]);
remeasureDropTargets({ group: "items" });
```

Use the narrowest input that matches the layout change. Remeasurement updates
cached target rectangles; it does not send a drag update by itself. If the active
target or sortable preview should update before the next pointer or keyboard
movement, call `useRecomputeActiveDrag` after remeasurement.

## Cancel And Cleanup Behavior

Sortable preview movement is temporary. During a drag, the underlying DOM behavior may move the dragged DOM element to preview where it would land. On cancel, no-target release, invalid-target release, active drag reset, and successful drop, preview state is restored or cleared by the package.

React state remains yours. If a drag is canceled, you usually only clear app-owned active styling in `onDragEnd`. Do not manually undo the preview DOM movement.

`onDragEnd` receives:

```ts
type DragEndResult =
  | "dropped"
  | "no-target"
  | "invalid-target"
  | "canceled";
```

`onDrop` only fires for a valid successful drop. Use `onDragEnd` for cleanup such
as clearing active styling. Use `onDrop` for sortable order commits.

## Performance Notes

- Keep ids stable. Do not use array indexes as `draggableId` values for reorderable lists.
- Keep `onDragUpdate` cheap. It can run frequently, so avoid expensive state updates there.
- Commit durable order in `onDrop`, not during every pointer move.
- Avoid recreating ids, modifier arrays, and callback-heavy item structures unnecessarily.
- If you create modifier arrays for `DragProvider`, use `useMemo`.
- For large lists, avoid full React rerenders during drag when possible. The package includes sortable candidate narrowing in the underlying DOM behavior, but this is not a virtualization API.
- Remeasure only after layout changes that should affect targeting, then recompute
  when the active target should update immediately.

## Common Mistakes

- Do not use unstable ids.
- Do not treat `placementBoundary` as targeting.
- Do not commit order on every drag update.
- Do not forget `useDropContainer` for empty lists or board columns.
- Do not infer the destination container from `dropTargetId`. Use
  `sortablePlacement.containerId` when sortable placement is present.
- Do not rely on stale measurements after app-owned layout changes.
- Do not expect `useRemeasureDropTargets` to send a drag update by itself. Call
  `useRecomputeActiveDrag` when targeting should update immediately.
- Do not recreate item ids, modifier arrays, or expensive callbacks unnecessarily.
